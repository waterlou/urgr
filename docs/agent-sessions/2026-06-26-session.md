## Session 3: 2026-06-26 — rom_source_id prior-version fallback

## Session 4: 2026-06-26 — Pre-create platform dirs to fix intermittent IO error

### Goal
Fix intermittent "IO error: No such file or directory (os error 2)" during build, which the user suspects is caused by deferred `create_dir_all` (moved to just before `copy` in Session 1 to avoid creating empty neogeo dirs).

### Problem
When `create_dir_all` for the platform subdirectory happens lazily at copy time, there's a window where the directory doesn't exist yet but is expected by other code paths. On a fresh build, this manifests as transient IO errors that resolve on retry.

### Changes

#### Rust: rom-manager
- **`libs/rom-manager/src/builder.rs`**: Added a new step between Phase 4 (import index) and Phase 5 (copy loop) that pre-creates platform subdirectories for all games. For games with `rom_source_id` (neogeo→arcade), resolves to the source game's platform so empty `neogeo/` dirs are never created. Only platforms that actually appear in the current version's games are created.

## Session 5: 2026-06-26 — find_by_content returns stale paths from consumed source files

### Goal
Fix persistent "IO error: No such file or directory (os error 2)" during builds for MAME collection.

### Root Cause (two layers)

**Layer 1**: `find_by_content` at `builder.rs:257` returned `Some(new_path)` even when `std::fs::rename` failed — the renamed path didn't exist on disk. Fixed by checking `is_ok()`.

**Layer 2** (diagnosed with path-context wrapping): The import index (`crc_to_zips`/`name_to_path`) is built ONCE during `ImportIndex::scan` before the copy loop. When multiple games share the same CRC set (identical zip content), the first game's `find_by_content` renames the source file to its game name. The second game gets a stale path from the index — the file no longer exists at that name.

Example trace:
```
1. Game "jojobajr1": find_by_content finds "wrongname.zip" by CRC → renames to "jojobajr1.zip" → copy OK
2. Game "jojobaner2": find_by_content finds same stem "wrongname" in stale index → "wrongname.zip" doesn't exist → returns stale path → copy fails ENOENT
```

**Fix**: Before rename, check if `new_path` already exists (from a previous game's rename of the same zip) — use it directly. Check if `src_path` exists — if not, the file was consumed by a previous game, return `None`.

### Other fixes in this session
- `walk_files` made resilient to entries vanishing during iteration.
- All `std::fs::copy` and `create_dir_all` calls in the build loop wrapped with path-context error messages.
- Pre-create platform subdirectories (resolving `rom_source_id`) before main copy loop to avoid race conditions from deferred `create_dir_all`.
