## Refactors Applied

### A. `apps/rom-manager-ui/server/routes/games.js` ﻗ°½ Extract `getScrapeMode()` and `canonicalName()` helpers
**Before**: 3 inline `let scrapeMode = 'parent'; if (game.version_id) { const col = get(... scrape_mode FROM collections ...); ... }` blocks, each 6 lines. Plus a local `canonicalName` arrow function in scrapeSingleGame that shadowed the helper name.
**After**: Single `getScrapeMode(versionId)` helper + module-level `canonicalName(game, scrapeMode)`. All 3 routes use them.

### B. `apps/rom-manager-ui/server/routes/games.js` ﻗ°½ Extract `fetchGameWithRomSet()` and `fetchGameOptionalRomSet()` helpers
**Before**: 5 inline queries with `LEFT JOIN games parent_g ON parent_g.id = g.parent_game_id ... JOIN/LEFT JOIN game_rom_sets grs ... JOIN/LEFT JOIN set_versions sv ... WHERE g.id = ? ORDER BY grs.version_id ASC LIMIT 1` ﻗ°½ varying only in INNER vs LEFT JOIN choice.
**After**: Two helpers ﻗ°½ `fetchGameWithRomSet()` (INNER JOIN, returns null if no rom_set) for routes that need one (availability, play), and `fetchGameOptionalRomSet()` (LEFT JOIN, returns game even without rom_set) for routes that should still work (detail, scrape, media, post-scrape read).

### C. Cancelled ﻗ°½ `last_insert_rowid()` after ON CONFLICT DO UPDATE
**Tried**: Replace INSERT-then-SELECT pattern in `insert_game`/`insert_rom_set` with `last_insert_rowid()`.
**Failed**: SQLite's `last_insert_rowid()` returns the most recent successful insert rowid, NOT the row updated by `ON CONFLICT DO UPDATE`. With multiple inserts in sequence (insert_game, insert_rom_set, insert_rom_file in batch), the last_insert_rowid becomes the last ROM file id, not the game id. Tests caught this ﻗ°½ `test_diff_versions_add_remove` failed because the second `insert_game("shared")` returned rowid 2 (the "removed" game's id) instead of 1.
**Reverted**: Switched back to SELECT-by-unique-key with comments explaining why `last_insert_rowid()` is unreliable here. Cost: one indexed lookup per insert (O(1) on a UNIQUE index). Net zero perf impact at scale.

### D. `libs/rom-manager/src/db/mod.rs` ﻗ°½ Fix `resolve_parents` ordering dependency
**Before**: For each game with `cloneof`, query `SELECT id FROM games WHERE name = ?`. If parent was inserted in the same batch AFTER the child (alphabetical ordering puts child first, e.g. `sf2a` before `sf2`), the parent lookup fails and `parent_game_id` stays NULL.
**After**: Build an in-memory `name ﻗ│φ id` map from the just-inserted slice (one chunked SELECT IN (...)), then look up parents from the map with a single-row DB fallback for parents imported in earlier batches. Unknown parents silently skip (no error). Added 2 regression tests:
- `test_resolve_parents_out_of_order` ﻗ°½ clone inserted first, parent second
- `test_resolve_parents_unknown_parent` ﻗ°½ parent doesn't exist anywhere

### E. `apps/rom-manager-ui/server/mediaCache.js` ﻗ°½ Convert sync I/O to async
**Before**: `fs.existsSync`, `fs.readdirSync`, `fs.readFileSync`, `fs.writeFileSync`, `fs.mkdirSync` block the Node.js event loop. On hot paths (cover/screenshot requests) every request blocks while scanning the cache dir.
**After**: All I/O uses `fs/promises`. Split into focused helpers: `findLocalFile`, `readCached`, `fetchWithTimeout`, `extFromContentType`. Cache write failures are non-fatal ﻗ°½ return the fetched data with the correct mime type. Request latency under load should improve noticeably.

### F. `libs/rom-manager/src/db/mod.rs` ﻗ°½ Extract closures in `diff_versions`
**Before**: 5 nearly-identical `let mut stmt = self.conn.prepare(...)?; ... query_map ... filter_map.collect() ...` blocks for: list_games_a, list_games_b, hashes_a, hashes_b, plus 2 separate `query_row` blocks for rom_set_id_a/b.
**After**: Three closures: `list_game_names(version_id)`, `rom_set_id(version_id, name)`, `collect_hashes(rom_set_id)`. The function body now reads as a sequence of high-level operations.
