## Root Cause

`last_insert_rowid()` is a **per-connection counter** that tracks the most recent
successful INSERT (not UPDATE) from **any table** on the connection. With
`ON CONFLICT DO UPDATE`, the statement succeeds but the conflict path is treated
as an UPDATE, not an INSERT, so the counter is NOT updated.

The result: after `insert_rom_set` inserts a row into `game_rom_sets`, the counter
has moved to a different table's rowid space. A subsequent UPSERT into `games` then
returns the wrong id (2, the most recent insert in `game_rom_sets`).

## Why My Previous Investigation Was Wrong

In the prior session I claimed "tests caught the issue" and reverted without
understanding the actual SQLite behavior. The truth:
- I was right that `last_insert_rowid()` is wrong on the conflict path
- I was right that the test caught it
- But I didn't dig into WHY — I jumped to "SQLite is unreliable" which was a
  hand-wavy explanation. The actual mechanism is the counter tracking inserts
  from any table, which is documented but not obvious.

## The Fix: `RETURNING id`

SQLite ≥ 3.35 supports `INSERT ... RETURNING id` which returns the rowid of the
affected row whether it was inserted (new) or updated (conflict path). This is
the correct primitive for getting the id of an UPSERT result.

### Before
```rust
self.conn.execute(
    "INSERT INTO games ... ON CONFLICT(name) DO UPDATE SET ...",
    params![...],
)?;
let id: i64 = self.conn.query_row(
    "SELECT id FROM games WHERE name = ?1",
    params![game.name],
    |r| r.get(0),
)?;
```

### After
```rust
let id: i64 = self.conn.query_row(
    "INSERT INTO games ... ON CONFLICT(name) DO UPDATE SET ...
     RETURNING id",
    params![...],
    |r| r.get(0),
)?;
```
